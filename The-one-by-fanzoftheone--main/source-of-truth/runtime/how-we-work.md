# THE ONE — How We Work

## Build Method

We do not build random code.
We build through the ecosystem.

Every new suggestion must be mapped first to:

- lab
- module
- app
- marketplace
- warehouse
- companion
- integration
- agent

## Working Style

- strengthen existing flow
- prefer strongest connected path
- avoid drift
- avoid side systems that do not connect
- preserve revenue lane
- preserve growth lane

## What gets implemented first

1. Revenue lane mechanics
   - module registry
   - module install/export
   - builder generator
   - marketplace publishing
   - app generation

2. Platform intelligence
   - warehouse memory
   - agent runtime
   - backpack persistence
   - companion reflection and memory
   - pattern detection

## Change Rule

When changing code:

- use current repo structure
- do not change frameworks unless required
- do not replace working systems with unrelated ideas
- integrate improvements into existing flow

## Memory Rule

Store important working truth in repo docs, not only chat.

Source of truth lives in:

source-of-truth/

## Orchestrator Rule

User stays orchestrator.
Models assist, compare, test, and execute.
No hidden orchestrator replaces human control.

## Notes Rule

When a new strong idea appears:

- place it in the right bucket
- note where it belongs
- decide whether it is lab, module, or app
- then implement

## Final Rule

The One is built through connected systems, not isolated features.
